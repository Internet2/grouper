Grouper 0.5 Implementation Guide

 

This guide will help you to build and operate Grouper 0.5. As this is the initial release and there is as yet no previous experience with Grouper implementation, even by its developers, the initial form of this guide is far from a comprehensive treatment of the topic. Our strategy is to bootstrap the documentation process by providing enough information to enable early adopters to get going and subsequently contribute to the enhancement of this document by their experiences.

Along with obtaining Grouper 0.5 (see the MACE-Dir-Groups website http://middleware.internet2.edu/dir/groups for instructions), adopters should join the grouper-users at internet2 dot edu and possibly the grouper-dev at internet2 dot edu mailists for collective self-help with implementation and to discuss ideas for further development of Grouper. Instructions for joining these lists are also on the MACE-Dir-Groups website.

1.               Introduction to Grouper 0.5

Grouper 0.5 provides an API supporting basic groups management capabilities - creation, update, and removal of groups from a Groups Registry, subgroups, limited querying capabilities, and logging. It also includes sample programs illustrating how to load data into and extract data from the Groups Registry.

In Grouper Phase 1 Specifications the components & architecture section depicts the major components of Grouper and their relation to elements of an implementation sites existing identity management infrastructure. Those components are an API, a relational database called the Groups Registry, a set of aging tasks, and one or more user interfaces for managing groups. Grouper 0.5 is comprised of an alpha release of the API and procedures to create the Groups Registry within an HSQLDB database. The user interface, aging tasks, and some substantial elements of API functionality are slated for future release according to the following table.

Version

0.6

0.7

In section 2 of this document well start by describing how to build, configure, and install the API and Groups Registry. Well lay out Groupers information and privilege models in section 3 and consider how they may integrate with existing identifier namespaces and extend existing identity management practices. In section 4 well consider, at a functional level, how Grouper may be made to articulate with an existing infrastructure.

In addition to this Implementation Guide, the Grouper 0.5 distribution contains javadoc and other germinal guides for developers (i.e., good intention outwieghs substance at the moment). All of these are found in the doc/ directory of the distribution, which also contains a file called NEWS that lists features in 0.5, those that are slated for a future release, and known problems with 0.5. If this document and the NEWS file appear to be in conflict on any point, the NEWS file should be considered authoritative.

Perhaps the best source of API usage guidelines for developers at the moment is the set of contributed programs and the suite of test programs included with the distribution. They exercise many functional aspects of the API, and the latter serve as the benchmark by which to assess that a Grouper installation is working properly.

"Grouper Phase 1 Specifications" contains an Entity Relationship Diagram for an early form of the Groups Registry schema. An ERD of the schema deployed in Grouper 0.5 can be found here.

In the rest of this document italics are used when introducing or defining terms and for file, method, class, and table names.

2.               Building Grouper 0.5

The top-level directory of the unpacked Grouper 0.5 distribution contains the following directories and files:

contrib/ contributed software

doc/ nascent document repository

javadoc/ javadoc for the Grouper 0.5 API

java/

tests/ test programs

sql/ sql scripts for creating, initializing, and testing the Groupers Registry

LICENSE license under which Grouper 0.5 may be used

build.xml ant configuration file

The file java/lib/README lists the 3rd party software used by grouper 0.5 and identifies the version, source, and license for each. README files occur here and there you should read them!

Well use the symbol "$GROUPER_BASE" below to refer to the top level directory of your Grouper distribution.

2.1.        Build instructions

1.         Get the latest Java SDK (from http://java.sun.com/j2se). The JRE alone is not sufficient.

2.         Get ant (from http://ant.apache.org).

Directions for installing ant are in the unpacked distribution at docs/manual/install.html#installing. You may need to copy tools.jar from $JAVA_HOME/lib to $ANT_HOME/lib to get ant working properly.

3.         Change directory to the root of the unpacked Grouper 0.5 distribution.

4.         Do "ant build".

This will compile grouper and place the class files under the $GROUPER_BASE/build directory.

5.         Do "any db-init db-init-test" to create and test the Groups Registry in the HSQLDB database provided. The database is created in the $GROUPER_BASE/dist/run/ directory.

1.      A "HSQL Database Manager" window will pop up with some commands preloaded. Press "Execute", wait for the result, then exit the HSQL Database Manager.

2.      A second HSQL Database Manager window will pop up with some more commands preloaded, but this time youll see a list of tables in the Grouper Registry in the left-hand pane of the window. Press "Execute". At this point the Groups Registry has been seeded with definitions of the "base" and "naming" group types (in the grouper_type, grouper_type_def, and grouper_field tables) and the "GrouperSystem" subject has been created in the grouper_subject table. You can execute select statements to see these, just for fun. Exit the HSQL Database Manager when youre done.

3.      A third HSQL Database Manager window will pop up with yet more commands preloaded. These will test that the HSQLDB instance is in fact working by entering a few test values into the grouper_subject table. Press the "Execute" button. Exit the HSQL Database Manager when youre done.

6.         Do "ant test". This compiles and runs a suite of test programs that exercise the API to ensure that alls well. The testing log is in the $GROUPER_BASE/dist/run/ directory.

At any point you can run "ant db" in the $GROUPER_BASE directory to fire up the HSQL Database Manager app to look at whats in the Groups Registry. Youll have full access to the database when you do.

7.         Do "ant dist" to create two jar files containing all of Groupers components and the 3rd party libraries on which it depends, located in $GROUPER_BASE/dist/lib. This will ease classpath formation for running Grouper from locations outside of $GROUPER_BASE.

2.2.        Configuring Grouper

Grouper 0.5 has only a few configuration options. The configuration files are in the $GROUPER_BASE/conf directory. That directory contains five files, two that should not be modified and three that should be. Do not modify ehcache.xml and Grouper.hbm.xml. Do modify Grouper.properties, hibernate.properties, and log4j.properties.

The Grouper.properties file for version 0.5 contains just three declarations. First is the subjectID for Groupers superuser, by default GrouperSystem. This subject has unlimited prviliege within the Grouper API. The other two declarations specify the classes implementing Groupers Access and Naming Privilege Interfaces. These are defaulted to use Groupers internal privilege implementation.

The hibernate.properties and log4j.properties must be changed to reflect the location of each Grouper installation. In conf/hibernate.properties, change:

hibernate.connection.url = jdbc:hsqldb:/FULL/PATH/TO/dist/run/grouper

 

And in conf/log4j.properties, change:

log4j.appender.rfile.File = /FULL/PATH/TO/dist/run/grouper.log

Other declarations in those files may also be changed to reflect desired logging characteristics or use of an alternate RDBMS.

2.3.        Running Grouper

To run a java program using the Grouper 0.5 API, the java environment must have a classpath that includes:

      full pathname of the Grouper.jar file produced by "ant dist"

      full pathname of the Grouper-lib.jar file produced by "ant dist", and

      full pathname of the directory containing Grouper's configuration files.

2.4.        Building and running sample applications

Grouper 0.5 includes four sample programs to aid in getting started with a Grouper deployment.

2.4.1.        Sample subject loader - csv2subject

The out-of-the-box method for providing Grouper 0.5 with subjects, i.e., identifiers for people or other objects that are potential group members, is to load them in to the grouper_subject table. The subject loader is a very simple program for so doing. It takes a comma separated list of subjectID, subjectTypeID and loads them directly into the table. Subjects and the preferred means of dealing with them are discussed in sections 3.6 and 4.3 below.

The $GROUPER_BASE/contrib/load/csv2subject directory has a simple subject loader and a sample csv file for loading subjects into a test database. To use it,

1.         change to the $GROUPER_BASE/contrib/load/csv2subject directory

2.         read the README file and follow its instructions.

csv2subject uses JDBC to communicate directly with the database, i.e. it does not use the Grouper API. Its configuration file, csv2subject.properties, will need to be modified to work with a different database product (see section 4.1).

2.4.2.        Sample group loader csv2group

The csv2group program is a very simple means to add namespaces, groups, and members to the Groups Registry.However, it doesn't delete groups or memberships - that is an exercise left to each site's developers! To use it,

1.         change to the $GROUPER_BASE/contrib/load/csv2group directory

2.         read the README file and follow its instructions.

The csv2group program interacts with the Groups Registry as the GrouperSystem subject. You'll want to add the subjects in the csv2subject/sample.csv file before loading the sample groups in this sample.csv the members of the sample groups are taken from the list of sample subjects.

2.4.3.        Sample group manager groupmgr

The groupmgr program provides a command line interface to add namespaces to the Groups Registry and to add and delete groups and memberships. Unlike csv2group, you can specify which subject groupmgr should act as. To use it,

1.         change to the $GROUPER_BASE/contrib/load/groupmgr directory

2.         read the README file and follow its instructions.

2.4.4.        Sample query tool grouperq

The grouperq program provides a command line interface for performing simple queries on the Groups Registry. It is also useful for "running as" a given subject as a means of verifying the subject's access privileges. To use it,

1.         change to the $GROUPER_BASE/contrib/load/groupmgr directory

2.         read the README file and follow its instructions.

3.               Using Groupers data & privilege models

Grouper works by managing two principal types of objects: groups and namespaces. A group is a list of members and other associated data that can be referred to in several ways. A namespace is a context in which a group is named. From the perspective of information management, a Grouper deployment is defined by which people and applications have what access to each namespace and group. In this section we present Groupers data and privilege models and discuss several aspects of a Grouper deployment: naming, distributing management of groups, and choice of identifiers for members.

3.1.        Grouper 0.5 fields

In Grouper parlance, each group has a set of fields, and each field can either be a list or an attribute. In Grouper 0.5, a list is a multivalued set of subjects and an attribute is a single-valued string. Each subject has a single subjectType, the full set of which is determined by each implementation site. 3.2.        Extending Groupers fields

To support some of the use cases identified by MACE-Dir-Groups working group members, Grouper was designed from the start to enable implementation sites to extend the set of attributes and lists available to associate with some groups. This was done by declaring attributes, lists, and their properties in tables within the Groups Registry rather than having them repesented as static schematic elements of that database. Sets of attributes and lists are packaged into group types, and, in a future release of Grouper (do we know which?), each group can be assigned to any number of types. This is similar to how schemas for LDAP directories are managed and extended: LDAP attributes are packaged into objectclasses, and each LDAP entry can have a set of objectclasses associated with it, allowing different entries to serve different purposes. However, since in Grouper 0.5 a group can have only a single type, each site using extended types will need to ensure that each extended type includes all of the naming related base fields, members, and all of the privilege lists unless the site is using an access privilege interface implementation different from Groupers default.

Grouper 0.5 ships with two group types. All groups are of the base type - its structure is given in Table 1. Namespaces are modelled internally to Grouper as groups of naming type. Its structure is given in Table 2.

Although the API provides no support for managing group types, a site may alter or learn by example from the Groups Registry initialization script $GROUPER_BASE/sql/init.sql to populate the grouper_type, grouper_typedef, and grouper_field tables to determine the set of attributes, lists, and group types they want in their Grouper deployment.

Note: the guid attribute is an operational attribute. It is stored in the database as the Grouper_group:groupID column and exposed in the API with the GrouperGroup.id method instead of with the GrouperGroup.attribute method. In presentation and documentation contexts well use the term guid.

3.3.        Naming elements & examples

Table 1 and Table 2 show that groups and namespaces have the same naming attributes with the same semantics. Groups and namespaces are created and named within namespaces, and namespaces can be arranged in a flat or hierarchical style, as desired by each implementation site.

Since a group must be named within a namespace, it is necessary to create a namespace before any groups can be created. In Grouper 0.5 namespaces are modelled as groups of naming type, so namespaces and groups are created in a similar manner, using the GrouperGroup.create method. Groups are created using a form of the create method that defaults to creating groups of the default, base, type. Namespaces are created using a form of the create method in which a non-default type is explicitly declared. For example, the following code fragment creates a top-level namespace named "uofc" and then creates a group named "exec_council" within the "uofc" namespace.

 

String stem = Grouper.NS_ROOT;

String extn = "uofc";

GrouperGroup ns = GrouperGroup.create(s, stem, extn, Grouper.NS_TYPE);

 

String stem = "uofc";

String extn = "exec_council";

GrouperGroup g = GrouperGroup.create(s, stem, extn);

 

A hierarchy delimiter is used in string representations of the names of groups and namespaces. The delimiter is a ":" character in Grouper 0.5, but in the next release it will be a configurable choice within a constrained set of characters. Continuing the example above, the name of the "exec_council" group within the "uofc" namespace is "uofc:exec_council".

Each newly created group or namespace contributes just one new element, the extension, to its accumulative name. DisplayNames of groups and namespaces are likewise accumulative thats why the name and displayName attributes are system maintained and read-only. 3.4.        Grouper 0.5 privileges

There are two categories of privileges in Grouper. Access privileges determine what a subject can do to a group. Naming privileges determine what a subject can do to a namespace. These are listed in the following table.

Grouper filters group loads and search hits based on a subjects VIEW and READ privileges for each matched group. Subjects lacking VIEW privilege to a group will not have that groups existence revealed to them by the Grouper API. Subjects with VIEW but lacking READ can read the groups naming attributes and no others. OPTOUT privilege to a group does not enable the subject to remove themselves from effective membership in the group, only immediate membership. The creator of a group is automatically granted ADMIN privilege to the group. The creator of a namespace is automatically granted STEM privilege to the namespace. Although STEM privilege does not imply CREATE privilege, a subject with STEM privilege can assign themselves CREATE privilege.

Caveat: Some of the above privilege characteristics will be fully implemented only in the next release, a maintenance release to be called Grouper 0.5.1.

Top-level namespaces, i.e., those for which the stem attribute is null, can only be created by the GrouperSystem distinguished subject in Grouper 0.5.

The Grouper API presents and itself uses abstracted interfaces, GrouperAccess and GrouperNaming, to manage access and naming privileges. This design permits implementation sites to replace Groupers default implementation of either or both of these interfaces with ones of their own devise, in case the site has or will have privilege management capabilities with which they wish to integrate management of Grouper privileges (see section 4.5).

The privilege interfaces permit privileges to be assigned to subjects of any type, including groups. An implementation of the privilege interfaces that relies on Grouper to resolve subjects that are groups can take advantage of Groupers subgroup, compound group, and aging capabilities to enrich privilege management. In particular, Groupers default privilege implementation uses group management to support inheritance and aging of privileges.

Although it is not possible to extend Groupers privilege model, sites that extend the set of attributes and lists in their deployment can declare which privileges are needed to read and to write each attribute and list. These declarations are in the grouper_field table. Although the API provides no support for managing extended fields or their security, sites may alter or learn by example from the Groups Registry initialization script $GROUPER_BASE/sql/init.sql to do so.

3.5.        Using privileges to distribute group management capabilities

One of the leading reasons why there is interest in Grouper is that many sites recognize a need to involve many sources of authority in managing resource access, distribution lists, and application customization the types of needs which groups help to satisfy. Groupers privilege model supports five ways to distribute management of groups.

1.         Create a group and assign someone UPDATE privilege to it. This delegates the ability to maintain the groups membership.

2.         Create a group and assign someone ADMIN privilege to it. This delegates the ability to manage who can maintain the groups membership as well as manage all access to the group.

3.         Create a namespace and assign someone CREATE privilege to it. This delegates the ability to preside over the collection of all groups they create in that namespace.

4.         Create a namespace and assign someone STEM privilege to it. They can rule over an entire hierarchy of namespaces and groups, all sharing a common root namespace.

5.         Create a group and assign OPTIN and/or OPTOUT privilege to other groups. Subjects belonging to those other groups are entitled to manage their own membership in the group.

An example of an actual scenario for delegation of group management to solve an access control problem can be found in http://home.uchicago.edu/~tbarton/camp-200411/20041117-authz.ppt.

3.6.        Subject identifiers

Most often the members of your groups will be people and other groups. But they might also be of any other type (applications or computers, for example) for which you now or eventually will manage identities. Grouper 0.5 supports a site-defined set of subjectTypes (see section 4.3 below).

Identifiers and their management can get complicated. They can be revoked or not, re-assigned or not, lucent or opaque, etc. Depending on such characteristics, a given identifier might be a good or bad choice to use in the context of managing the identified subjects group memberships.

For example, a username is often lucent easily remembered by the person to whom it is associated. But it may also be revokable, meaning that it no longer refers to that person (perhaps they have a new one), or even re-assignable, meaning that it might refer to some other person at a later time. If a username is used to record membership, username changes must trigger corresponding membership changes. A username is better suited to authentication than it is to indicating membership.

On the other hand, an opaque registryID (machine, not human, readable) that never changes is great for membership, but lousy for authentication it might not even be known by the person to whom it is associated. How would I identify myself to Grouper if I wished to opt-in to a list or manage a group?

Grouper accomodates subject identifer issues in two ways. First, it maintains guids for every subject and group in the deployment. These are never exposed by the API, but are associated with externally supplied subject identifiers in the grouper_member table. This approach allows the identifier associated with a given subject to be changed without any need to change actual memberships (although the supporting GrouperMember method is only slated for inclusion in the 0.5.1 release).

Second, by release 0.6, the Grouper API will incorporate an identifier lookup and translation service called the Subject Interface being co-developed by the Grouper and Signet projects. The Subject Interface is used to lookup subjects that are presented with an identifier in one namespace and obtain identifiers in other namespaces for that subject. That means that it can translate a username into a registryID and vice-versa, for example. So, when a user authenticates to an application using the Grouper API, that application can use the Subject Interface to fetch an identifer for that person chosen by the site for use in memberships. Similarly, when a membership in the Groups Registry is to be expressed elsewhere, the identifer used for group members can be translated by a provisoning connector by use of the Subject Interface into one that is suitable in the provisioned context.

4.               Integration Tasks

Things youll need to do to integrate grouper with your infrastructure and practices.

4.1.        Using your own RDBMS

Grouper 0.5 ships with the HSQLDB relational database and Hibernate (http://www.hibernate.org/) for object persistence. Hibernate uses JDBC to actually map objects to persistent storage. So it should not be difficult to make Grouper use any of a variety of common RDBMS systems.

To use an alternate database youll need to

1.      amend Groupers Hibernate configuration file to connect to your RDBMS, and

2.      arrange an alternative database initialization process.

The Hibernate configuration is in $GROUPER_BASE/conf/hibernate.properties. Several of its declarations will need to be modified. We intend to add subsections at this point in the Implementation Guide detailing how various databases have been connected with Grouper based on the experiences of early adopters.

The "db-init" ant target in $GROUPER_BASE/build.xml creates and initializes the database relying on SQL scripts located in the $GROUPER_BASE/sql/ directory. These are believed and intended to be vanilla SQL92 and if so ought to work for other relational databases, but this has not been verified as yet. Some modification may be required.

To verify that your database is functioning correctly with Grouper, change directory to $GROUPER_BASE and run "ant test".

4.3.        Getting subjects into Grouper

Section 3.6 broached issues surrounding the choice of subject identifiers for use with Grouper. Having made a choice, this section tells you how to get those identifiers integrated with Grouper. You may want to stare at the Groups Registry ERD while reading this section.

The grouper_subjectType table is used to declare what subjectTypes are in a Grouper deployment, and, for each type, the name of the adapterClass that will be called to lookup and load objects of each type. Grouper 0.5 installs with this table defaulted to support two subject types: person and group. The default person adapterClass, called SubjectTypeAdapterPersonImpl, provides sites with a bootstrap to getting subjects into Grouper 0.5. It just looks in the grouper_subject table to qualify that the subjectID supplied to the Grouper API is valid. So all youve got to do initially is populate the grouper_subject table with all of the subjectIDs encompassed by your Grouper deployment, which is what the csv2subject contributed program does (see section 2.4.1).

A better means of integrating your person type subjectIDs with Grouper 0.5 is to supply your own adapterClass for the person subjectType. This class will need to validate and create subject objects given a subjectID. So, for example, it might hit your LDAP directory or your Person Registry directly to validate that a subjectID is in fact one that is valid for a group management operation. This would remove the need to maintain the grouper_subject table its sole purpose in the Groups Registry is to bootstrap subjects into Grouper.

With implementation of the Subject Interface, anticipated for Grouper 0.6, a richer set of subject data obtained from your identity management operation is made available to implement identifier translation capabilities as described in section 3.6 and subject search capabilities in support of the Grouper UI. The default adapterClass for the person subjectType in Grouper 0.6 will support these identifier search and translation capabilities by reliance on the grouper_subjectAttribute table. That will allow you to bootstrap your Grouper deployment by maintaining this table until such time as you can supply your own person subjectType adapterClass with the requisite functionality.

NOTE: The adapterClass for the group subjectType should not be changed! Else why are you bothering with Grouper?

4.4.        Loaders and provisioning connectors

Loaders and provisioning connectors are defined in the components & architecture section in Grouper Phase 1 Specifications. Loaders are automation process that use the Grouper API to load and maintain the Groups Registry from other sources, and provisioning connectors are automation processes that use the Grouper API to extract information from the Groups Registry and represent it in other databases, directories, etc.

As this is the initial release, there is as yet no contributed code that has been used to get real data into or out of Grouper, although the contributed applications discussed in section 2.4 above could be used as a starting point for developing them. So this section will merely identify some of Groupers capabilities relative to loaders and provisioning connectors.

All interactions with the Grouper API are conducted within a session that is constrained by the privileges of the sessions subjectID, so each loader and provisioning connector will need to be assigned one. The default GrouperSystem subject can be used, or others may be assigned as necessary to meet site-specific security requirements.

The Grouper API does not authenticate the subjectID asserted by an API client. It is assumed that sites will secure access to the Grouper API as they do other infrastructure services, with a combination of host and application platform security measures.

Loaders and provisioning conectors may use the Subject Interface, after its incorporation into Grouper in release 0.6, to lookup and translate external identifiers for subjects into those selected for use as Grouper subjectIDs.

The GrouperQuery class encapsulates the search filters implemented in Grouper 0.5 for selecting groups. These include

      specifying a time before which the group was modifed

      specifying a time after which the group was modifed

      specifying a time before which the group was created

      specifying a time after which the group was created

      specifying a group type

      specifying whether the group has immediate or effective memberships, or both.

These filters can be combined to constrain a single search. A search is executed using the selected filters by the GrouperQuery.query method. Further seach capabilities are slated for later releases of Grouper. See Grouper API Search & Browse Capabilities for details.

Grouper 0.5 has no way to limit the number of objects returned by a search. That too is planned for a subsequent release.

GrouperQuery.query and the GrouperGroup and GrouperMember list* methods for listing immediate and effective memberships return lists of GrouperList objects. The GrouperList.group and GrouperList.member methods convert GrouperList objects into GrouperGroup or GrouperMember objects, respectively.

In the case of a list value that is a GrouperMember object representing an effective membership, the via group for that membership is returned by the GrouperList.via method. A via group is a group in which a member has an immediate membership that qualifies it for effective membership by virtue of a subgroup or compound group relationship between the via group and the group in which the member has effective membership (the via group concept is discussed in more detail in Grouper Phase 1 Specifications). If there's more than one via group qualifying a given member's effective membership in some group, then the list of GrouperList objects returned by GrouperGroup.listEffVals or GrouperMember.listEffVals contains a GrouperMember object for each via group. This allows API clients like loaders and provisioning connectors to determine which immediate memberships must be removed in order to make a given effective membership go away.

The modifyTime and modifySubject are updated with each change to any of the group's fields. A change in effective membership or effective privilege is not reflected in the values of these operational attributes. Such a capability has been requested for a future version of Grouper.

The memberKey referred to in this table is an internal globally unique identifier that Grouper assigns to each subject it comes into contact with. The GrouperGroup.createSubject and GrouperGroup.modifySubject methods return the subjectID corresponding to this memberKey.
