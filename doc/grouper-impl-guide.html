This guide will help you to build and operate Grouper. As Grouper is still early in its development cycle, there is little practical implementation experience and thus this guide is far from a comprehensive treatment of the topic. Our strategy is to bootstrap the documentation process by providing enough information for early adopters to begin using Grouper and to subsequently contribute to the enhancement of this document by their experiences.

Along with obtaining Grouper (see the MACE-Dir-Groups website for instructions), adopters should join the grouper-users and possibly the grouper-dev mailing lists for implementation assistance and to discuss the further development of Grouper.

1. Introduction To Grouper

Grouper is an open source toolkit for managing groups. It provides an API for supporting basic group management capabilities - creation, reading, updating and deletion of groups from a Groups Registry, a central information repository complementary to a site's Person Registry. Grouper also supports automated and manual maintenance of groups, subgroups, limited querying capabilities, support for multiple database backends and the ability to integrate with existing identity management (IDM) infrastructure.

In Grouper Phase 1 Specifications the components & architecture section depicts the major components of Grouper and their relation to an implementation site's existing identity management infrastructure. Those components are an API, a relational database called the Groups Registry and one or more user interfaces for managing groups. While the API and other design details have changed since the Grouper Phase 1 Specification was released, the architectural drawing remains accurate. Future Grouper releases will provide an improved and more feature-rich API, UI enhancements, aging tasks, loading and provisioning support and performance improvements. A broad roadmap of the Grouper development cycle is below:

Version Release Date Description
0.5 200412 Initial API release providing a subset of the Phase 1 Specification: basic group management by automation processes
0.5.5, 0.5.6 200504 Improvements to public API and internals
0.6 ~200509 Initial UI release, I2MI Subject interface for integration with existing IDM infrastructure, full privilege model implementation and improved querying capabilities
0.9 ~200511 API and UI support for new Grouper API; preparation for 1.0
1.0 ~200601 API and UI support for group math
1.1 ~200603 API and UI support for aging tasks

The rest of this document will describe the Groups Registry; how to build, configure and run Grouper; integrating Grouper with an existing infrastructure; extending Grouper and sample applications that can be used to bootstrap a site's Grouper implementation.

In addition to this implementation guide, the Grouper distribution contains javadoc ($GROUPER/doc/javadoc) that provides some information on using the API and a list of changes from prior releases ($GROUPER/doc/NEWS). If this guide and the NEWS file appear in conflict, NEWS should be considered authoritative. We use the symbol $GROUPER throughout this document to refer to the top level of your Grouper distribution.

The best source of practical information on using the Grouper API to can currently be found in the contributed programs ($GROUPER/contrib/) and Grouper's test suite ($GROUPER/java/tests/), both of which are included with the Grouper distribution. They exercise many functional aspects of the API and the latter serves as a means of assessing that a Grouper installation is working properly. A Grouper development guide and improved sample programs are under construction and will be available at a later date.

2. The Groups Registry

2.1 Groups And Namespaces

Grouper works by managing two principal types of objects: groups and namespaces. A group is a list of members and various attributes. A namespace is a context in which a group or a another namespace may be located and and named along with various attributes. Both groups and namespaces may have a number of different privileges assigned to them. From the perspective of information management a Grouper deployment is defined by which subjects (people, applications, groups) have what access to each namespace and group.

The Groups Registry uses a RDBMS to store information on namespaces, groups, memberships and possibly privileges. Grouper is currently known to work and tested with HSQLDB, PostgreSQL and Oracle. This is the current Grouper database schema ERD.

2.2 Group And Namespace Attributes

Attributes are predefined single-valued pairs that may be assigned to both groups and namespaces. The names of attributes are case sensitive while the values are typically not.

Groups and namespaces have the following set of attributes:

Attribute Semantics
guid A system-assigned, read-only, globally unique identifier that is never reassigned.
stem The namespace in which the group or namespace is located. All groups must exist within a namespace. Stems may either exist at the root-level or within another namespace.
extension Relative name of the group or namespace within its parent namespace.
name A system-maintained, read-only string representation of the logical ordered pair of (stem, extension), used to facilitate searching for groups by name. The string representation of the name attribute is stem:extension.
displayExtension An alternative string representation for the group or namespace's extension attribute. The value defaults to the value of extension and will revert to the value of extension if deleted.
displayName A system-maintained, read-only string representation of the logical ordered pair of (displayName of stem, displayExtension), used to facilitate searching for groups by displayed name. The string representation of the displayName attribute is displayName of stem:displayExtension.
description Semantics defined by the implementation site.

Table 1 - Group And Namespace Attributes

In addition, each group and namespace in the registry has several operational attributes associated with it. These attributes are system-maintained and are not editable. The operational attributes are:

Operational Attribute Semantics
createTime Time at which the group was created.
createSubject The subject that created the group.
createSource Currently undefined.
modifyTime Time at which the group was last modified.
modifySubject The subject that last modified the group.
modifySource Currently undefined.

Table 2 - Group And Namespace Operational Attributes

The modifyTime and modifySubject are updated with each change to any of the group's attributes or lists. A change in effective membership or effective privilege is not reflected in the values of these operational attributes. A request has been made to include that ability in a future release of Grouper.

2.3 Group Lists

In addition to the attributes described above, groups (but not namespaces) also have a list of members.

List Semantics
members All members of the group. A group can have as a member any subject that can by located by the I2MI Subject sources that you have configured for your Grouper installation. This means that people, applications and groups may all be members of a group. If a group is made a member of another group, the members of the group will also be made members.

2.4 Naming Examples

The attributes table shows that groups and namespaces have the same naming attributes (stem, extension, name, displayExtension and displayName) with the same semantics. Groups and namespaces are created and named within namespaces and namespaces can be arranged in either a flat or hierarchical style as desired by the implementation site.

Since a group must be named within a namespace, it is necessary to create a namespace before any groups can be created. Namespaces are currently modeled internally as groups of naming type. The following code example demonstrates the creation of a namespace and a group within that namespace: 

  /*
   * Create a root-level namespace with the extension "uofc".
   */
  String        ns_extn   = "uofc";
  GrouperStem   ns        = GrouperStem.create(s, ns_extn);

  /*
   * Create a group with the extension "exec_council" within the "uofc"
   * namespace.  
   */
  String        grp_stem  = "uofc";
  String        grp_extn  = "exec_council";
  GrouperGroup  g         = GrouperGroup.create(s, grp_stem, grp_extn);

A hierarchy delimiter is used in string representations of the names of groups and namespaces. This delimiter is placed between every stem and extension. The delimiter is a ":" and this character can not be used as part of the extension for either namespaces or groups.

Each newly created group or namespace contributes just one new element, the extension, to its accumulative name. The displayName of groups and namespaces is likewise accumulative - that's why the name and displayName attributes are system maintained and read-only. Here are some further examples illustrating all of the naming attributes of several namespaces and groups.

Continuing with the example of the namespace and group created above, below are the attributes values that each would have:

namespace uofc
stem  
extension uofc
displayExtension uofc
name uofc
displayName uofc

group uofc:exec_council
stem uofc
extension exec_council
displayExtension exec_council
name uofc:exec_council
displayName uofc:exec_council

If we then assign the namespace a displayExtension value of "The University Of Chicago", the namespace and group will have the following attribute values:

namespace uofc
stem  
extension uofc
displayExtension The University Of Chicago
name uofc
displayName The University Of Chicago

group uofc:exec_council
stem uofc
extension exec_council
displayExtension exec_council
name uofc:exec_council
displayName The University Of Chicago:exec_council

Next we'll give the group a displayExtension value of "Executive Council".

namespace uofc
stem  
extension uofc
displayExtension The University Of Chicago
name uofc
displayName The University Of Chicago

group uofc:exec_council
stem uofc
extension exec_council
displayExtension Executive Council
name uofc:exec_council
displayName The University Of Chicago:Executive Council

Display values propagate down through subordinate namespaces as well:

namespace uofc:bsd
stem uofc
extension bsd
displayExtension Biological Sciences Division
name uofc:bsd
displayName The University Of Chicago:Biological Sciences Division

group uofc:bsd:eis_staff
stem uofc:bsd
extension eis_staff
displayExtension Enterprise Information Systems staff
name uofc:bsd:eis_staff
displayName The University Of Chicago:Biological Sciences Division:Enterprise Information Systems staff

This set of examples illuminates at least two things. First, that it would be a good idea for implementation sites to establish conventions for the assignment of naming attributes. An example of such is here (note that it was created before Grouper's design encompassed displayNames and at that time the term descriptor was used where we now say extension).

Second, that the names of groups, as with the titles of many types of things, influence people's initial expectations of the meaning of those groups. Seeing "Enterprise Information Systems staff" by itself gives a different impression than seeing it in the context of "Biological Sciences Division". That, in part, motivates the care taken in the design of Grouper's privilege model.

2.5 Privileges

There are two categories of privileges in Grouper: Access and Naming. Access privileges determine what a subject can do to a group. Naming privileges determine what a subject can do to a namespace. All Grouper API methods are subject to the Grouper privilege model and privileges can be assigned to subjects of any type, including groups. See Integrating Privileges for more information on the Grouper privilege model.

The access and naming privileges are listed in the following tables:

Privilege Semantics
VIEW Can see the group and can refer to it, e.g., to make it a subgroup of another group.

NOTE: If no explicit VIEW privileges have been assigned to a group, all subjects can VIEW the group.
READ Can read the group's public attributes and members list.

Implies VIEW.

NOTE: If no explicit READ privileges have been assigned to a group, all subjects can READ the group's public fields.
UPDATE Can update the group's members list and can grant and revoke OPTIN and OPTOUT privileges for this group.

Implies VIEW, OPTIN and OPTOUT.
ADMIN Can modify all group fields, can grant and revoke all access privileges, can rename the group and can delete the group. The creator of a group is automatically granted this privilege.

Implies all other access privileges.
OPTIN Can add oneself to the members list of the group.

Implies VIEW.
OPTOUT Can remove oneself from the members list of the group.

Implies VIEW.

NOTE: Can only remove immediate memberships.

Table 4 - Access Privileges

Privilege Semantics
CREATE Can create a group in the namespace.
STEM Can grant and revoke CREATE and STEM privilege for the namespace, can create a subordinate namespace, can delete the namespace and can modify all namespace fields. The creator of a namespace is automatically granted this privilege.

Top-level namespaces (those existing at the root of the Groups Registry and thus not possessing a parent namespace) can only be created by the GrouperSystem subject.

NOTE: This privilege does not imply CREATE but a subject with this privilege can grant themselves that privilege.

4. Building Grouper

If you intend to build and test Grouper using a RDBMS other than the default HSQLDB backend, please review the configuration sections for the database and subjects first.

  1. Get the Java SDK.

    The SDK may be downloaded from <http://java.sun.com/j2se/>. The JRE alone is not sufficient.

  2. Get Ant

    Ant may be downloaded from <http://ant.apache.org>.

    Directions for installing Ant are in the unpacked distribution at docs/manual/install.html#installing. You may need to copy tools.jar from $JAVA_HOME/lib to $ANT_HOME/lib to get ant working properly.

  3. Change directory to $GROUPER.

  4. ant build

    This will compile Grouper and place the class files under the $GROUPER/build directory.

  5. ant db-init

    This will create a Groups Registry in $GROUPER/dist/run/ directory using the default HSQLDB RDBMS.

  6. ant test

    This will compile and run the Grouper test suite to exercise the API and ensure that Grouper is properly configured.

    Note: Running the test suite is destructive and will delete all groups and memberships in the Groups Registry. Do not run the test suite against production data.

  7. ant db [optional]

    This will launch HSQLDB's GUI database manager. From within the manager you can run SQL queries against the Groups Registry.

  8. ant dist [optional]

    This creates two .jar files within $GROUPER/dist/lib/. One file contains all of the Grouper class files while the other contains all of the third party .jar files used by Grouper. This isn't required but does ease $CLASSPATH configuration.

5. Configuring Grouper

Grouper requires very little configuration by default. In fact, if a site is comfortable using the HSQLDB database and running all programs that interact with the Groups Registry from the $GROUPER directory, no configuration (outside of setting one's classpath) is required. The next sections will explain how to move beyond that specific (and default) setup.

5.1 Database

Grouper uses Hibernate to persist objects in the Groups Registry. As Hibernate uses JDBC to speak to a RDBMS, Grouper should be able to work with your preferred database backend provided there is a JDBC driver for it. Grouper is currently being tested against HSQLDB (the default), PostgreSQL and Oracle.

5.1.1 HSQLDB

Grouper uses HSQLDB as its default RDBMS. The ony configuration that may be required is to provide an absolute path to one's database by setting hibernate.connection.url in $GROUPER/conf/grouper.properties.

5.1.2 PostgreSQL

PostgreSQL is a supported Grouper backend and should only require small configuration changes to use. Grouper also includes a PostgreSQL JDBC driver; see $GROUPER/java/lib/README for version information. To use PostgreSQL you will need to:

  1. Uncomment the PostgreSQL section of $GROUPER/conf/hibernate.properties and modify as appropriate.
  2. Uncomment the PostgreSQL section of $GROUPER/conf/sources.xml and modify as appropriate.
  3. Set sql.type to postgres in $GROUPER/build.xml

5.1.3 Oracle

Oracle is a supported Grouper backend and should only require small configuration changes to use. An Oracle JDBC driver is not currently distributed with Grouper. To use Oracle you will need to:

  1. Uncomment the Oracle section of $GROUPER/conf/hibernate.properties and modify as appropriate.
  2. Uncomment the Oracle section of $GROUPER/conf/sources.xml and modify as appropriate.
  3. Set sql.type to oracle in $GROUPER/build.xml

5.1.4 Using Other Databases

While Grouper should work with other databases, it has not yet been tested with them nor does it provide support for them as shipped. To use an alternate database you will need to:

  1. Modify Grouper's Hibernate configuration file ($GROUPER/conf/hibernate.properties) to connect to your RDBMS
  2. Modify Grouper's Subject API configuration file ($GROUPER/conf/sources.xml) to connect to your RDBMS
  3. Create $GROUPER/sql/schema-$YOUR_DB_TYPE.sql
  4. Create $GROUPER/sql/test-data-$YOUR_DB_TYPE.sql
  5. Set sql.type to $YOUR_DB_TYPE in $GROUPER/build.xml
If you do use an alternate database that is not natively supported by Grouper, please contact the grouper-dev list with your results and schema files.

5.2 Subjects

See Integrating Subjects for more information on subjects and the I2MI Subject API.

The configuration file for Grouper's I2MI Subject API resolver is $GROUPER/config/sources.xml. The I2MI Subject API is used to map an external IDM infrastructure to Grouper internals and can be used to identify people, groups and applications from multiple sources of authority.

Grouper's default configuration uses Grouper as the source of authority for subjects of type group and several tables within the Groups Registry for subjects of type person. If you wish to use additional or different sources, you will need to adjust this file. At this time, the javadoc provided with each source adapter is probably the best location for configuration information.

Sites should not change the source for subjects of type group.

5.3 Privileges

$GROUPER/conf/grouper.properties must be modified if a site is using an external privilege resolver. Modify the interface.access or interface.naming configuration parameters to reflect the class name of the external privilege resolver that is being used.

5.4 Logging

Unless a site is running all programs from within $GROUPER the log4j.appender.rfile.File parameter in $GROUPER/conf/log4j.properties should be modified to use an absolute path to Grouper's logfile. Other parameters may be adjusted within this file to affect the verbosity and type of logging that Grouper produces.

5.5 Cache

ehcache is used by Hibernate to cache objects and query results to both memory and disk. Depending upon your usage and the size of your Groups Registry, you may want to adjust your cache settings. See the ehcache documentation for more details and then update $GROUPER/ehcache.xml as appropriate.

6. Running Grouper

As Grouper is currently just a database and a Java API for exercising that database, there is no "Groups Registry server" that needs to be running to interact with the Groups Registry. All that is necessary is that one be able to connect to your configured database and that the following items be in your classpath:

  • full pathname of the Grouper.jar file produced by ant dist
  • full pathname of the Grouper-lib.jar file produced by ant dist
  • full pathname of the directory containing Grouper's configuration files.

7. Integrating Grouper

This section is intended to demonstrate how to integrate Grouper with your existing infrastructure and practices. Unfortunately, because of where Grouper is in its development process, integration information remains sparse.

7.1 Loaders And Provisioning Connectors

Loaders and provisioning connectors are defined in the components & architecture section in Grouper Phase 1 Specifications. Loaders are automation process that use the Grouper API to load and maintain the Groups Registry from other source. Provisioning connectors are automation processes that use the Grouper API to extract information from the Groups Registry in order to represent it in other databases, directories, etc.

As this is an early release, there is as yet no real world contributed code that has been used to get data in and out of Grouper. However, the sample applications discussed later in this document could be used as a starting point.

All interactions with the Grouper API are conducted within a session that is constrained by the privileges of the session's subject. The default GrouperSystem subject may be used although it may be preferable to assign each loader and provisioning connector a unique subject to better meet site-specific security requirements.

When a new Grouper session is started, the API verifies the subject's existence but does not authenticate the subject. It is assumed that sites will secure access to the Grouper API - as they do other infrastructure services - with a combination of of host and application platform security measures.

The Grouper API provides methods for searching the registry to select groups to load or provision. The search filters are provided by the GrouperQuery class. The list of available filters includes:

  • specifying a namespace scope for searching within
  • specifying a group name
  • specifying an arbitrary group attribute
  • specifying a time before which the group was modified
  • specifying a time after which the group was modified
  • specifying a time before which the group was created
  • specifying a time after which the group was created
  • specifying a group type
  • specifying whether the group has immediate or effective memberships, or both.

Further search capabilities are slated for later releases of Grouper. See Grouper API Search & Browse Capabilities for details.

Grouper currently has no way to limit the number of objects returned by a search. Search result limits will be added in a future release.

7.2 Subjects

Most often the members of your groups will be people and other groups but they might also be of any other type (applications or computers, for example) for which you now or eventually will manage identities. By using the I2MI Subject API as a resolver service, Grouper allows sites to use multiple external IDM sources of authority to identify group members.

Identifiers and their management can get complicated. They can be revoked or not, re-assigned or not, lucent or opaque, etc. Depending on such characteristics, a given identifier might be a good or bad choice to use in the context of managing the identified subject's group memberships.

For example, a username is often lucent - easily remembered by the person to whom it is associated. But it may also be revokable, meaning that it no longer refers to that person (perhaps they have a new one), or even re-assignable, meaning that it might refer to some other person at a later time. If a username is used to record membership, username changes must trigger corresponding membership changes. A username is better suited to authentication than it is to indicating membership.

On the other hand, an opaque registryID (machine, not human, readable) that never changes is great for membership, but lousy for authentication - it might not even be known by the person to whom it is associated. How would I identify myself to Grouper if I wished to opt-in to a list or manage a group?

Grouper accommodates subject identifier issues in two ways. First, it maintains guids for every subject and group within the Groups Registry. These are never exposed by the API, but are associated with externally supplied subject identifiers within the Groups Registry. This approach allows the identifier associated with a given subject to be changed without any need to change actual memberships (although the supporting method is not yet available via the Grouper API).

Second, by incorporating the Subject API Grouper is able to lookup subjects that are presented with an identifier in one namespace and obtain identifiers in other namespaces for that subject. That means that it can translate a username into a registryID and vice-versa, for example. So, when a user authenticates to an application using the Grouper API, that application can use the Subject API to fetch an identifier for that person chosen by the site for use in memberships. Similarly, when a membership in the Groups Registry is to be expressed elsewhere, the identifier used for group members can be translated by a provisioning connector by use of the Subject API into one that is suitable in the provisioned context.

While Grouper currently only includes support for two sources of authority for subject information, sites could write a source adapter class that would hit their LDAP directory or Person Registry directly to validate that a subjectID is in fact one that is valid for a group management operation. It is expected that the I2MI Subject API will include a LDAP source adapter in the near future.

7.3 Privileges

The Grouper API provides and uses abstracted interfaces (GrouperAccess and GrouperNaming) to access and manage the Access and Naming privileges. This design permits implementation sites to replace Grouper's default implementation of either or both of these interfaces if they desire to integrate with an already existing external privilege management system. It is anticipated that Grouper will provide support for integrating with a Signet Privilege Registry in the future.

The default implementations of the Access and Naming privileges that are used by Grouper uses group list values to represent assigned privileges. Because privileges are modeled as list values in these implementations one is able to take advantage of Grouper's list management capabilities. In particular, this means that one can use subgroups to provide for the interitance of privileges. In addition, one will be able to take advantage of group math and aging tasks to provide an even richer privilege management environment.

If a site desires to replace one or both of the default privilege implementations, see the GrouperAccess and GrouperNaming javadoc for information on the interface that must be implemented. In addition, the interface.access and interface.naming configuration parameters in $GROUPER/conf/grouper.properties will need to be adjusted to reflect the privilege implementation being used.

7.3.1 Using Privileges To Distribute Group Management Capabilities

One of the leading reasons why there is interest in Grouper is that many sites recognize a need to involve many sources of authority in managing resource access, distribution lists and application customization - the types of needs which groups help to satisfy. Grouper's privilege model supports five ways to distribute management of groups.

  1. Create a group and assign someone UPDATE privilege to it. This delegates the ability to maintain the group's membership.
  2. Create a group and assign someone ADMIN privilege to it. This delegates the ability to manage who can maintain the group's membership as well as manage all access to the group.
  3. Create a namespace and assign someone CREATE privilege to it. This delegates the ability to preside over the collection of all groups they create in that namespace.
  4. Create a namespace and assign someone STEM privilege to it. They can rule over an entire hierarchy of namespaces and groups, all sharing a common root namespace.
  5. Create a group and assign OPTIN and/or OPTOUT privilege to other groups. Subjects belonging to those other groups are entitled to manage their own membership in the group.

An example of an actual scenario for delegation of group management to solve an access control problem can be found in <http://home.uchicago.edu/~tbarton/camp -200411/20041117-authz.ppt.

8. Extending Grouper

Grouper does currently provide some ability for sites to extend Grouper's default behavior and types to better fit local needs. Currently group types (including attributes and lists) and privileges may be extended with varying degrees of difficulty.

8.1 Group Types

To support some of the use cases identified by MACE-Dir-Groups working group members, Grouper was designed from the start to enable implementation sites to extend the set of attributes and lists available to be associated with groups. This was done by declaring attributes, lists, and their properties in tables within the Groups Registry rather than having them represented as static schematic elements of that database. Sets of attributes and lists are packaged into group types. This is similar to how schemas for LDAP directories are managed and extended: LDAP attributes are packaged into object classes, and each LDAP entry can have a set of object classes associated with it, allowing different entries to serve different purposes.

Unfortunately, at this time Grouper only allows a single group type to be assigned to a group. Thus, if you are extending the Groups Registry by creating new group types you will currently need to first duplicate the definition of the base group type and then add any additional attributes and lists that you may desire. Grouper ships with one custom group type: naming. This type is used internally to model namespaces.

Although it is not currently possible to extend Grouper's supported group types, attributes and lists with the API, sites can create new group types, attributes and lists. To adjust the schema you will need to manually alter the grouper_type, grouper_typeDef and grouper_field tables within the Groups Registry to determine the set of attributes and lists that are associated with an extended group type. The $GROUPER/sql/init.sql file may provide some guidance on how to update those tables with your desired extensions.

8.2 Privileges

Although it is not currently possible to extend Grouper's privilege model with the API, sites that extend the set of attributes and lists in their deployment can declare which privileges are needed to read and write each attribute and list. To adjust the schema you will need to manually alter the grouper_field table within the Groups Registry. The $GROUPER/sql/init.sql file may provide some guidance on how to update this table with your desired additions.

9. Sample Applications

Grouper includes four sample programs that demonstrate how to use parts of the API and that can be used to bootstrap a Grouper deployment. A larger and better selection of sample programs should be available in the near future.

9.1 Subject Loader: csv2subject

The default source for person subjects is within the Groups Registry itself. csv2subject is a simple program for populating the appropriate tables with subject information from a .csv input file. See Configuring Subjects and Integrating Subjects for more information on subjects.

csv2subject is located in $GROUPER/contrib/load/csv2subject. This program reads in a .csv file containing subjectIDs and subjectTypeIDs and then inserts the subjects into the Groups Registry via a direct JDBC connection. csv2subject uses Grouper's Hibernate configuration file for its JDBC configuration.

A sample .csv file is located in the $GROUPER/contrib/load/csv2subject directory.

To use csv2subject:

  1. Change to the $GROUPER/contrib/load/csv2subject directory
  2. Follow the instructions in the README file

9.2 Group Loader: csv2group

csv2group is a simple program for adding namespaces, groups and members to the Groups Registry. To use it:

  1. Change to the $GROUPER/contrib/load/csv2group directory
  2. Follow the instructions in the README file

A sample .csv file that creates a number of different namespaces, groups and memberships is located in this directory. To use the sample file you will first need to load sample subjects into the Groups Registry using csv2subject. The sample subjects are located in $GROUPER/contrib/load/csv2subject/sample.csv.

9.3 Group Manager: groupmgr

groupmgr is a program that provides a command line interface for manipulating the Groups Registry. It can add and delete namespaces, groups and memberships. To use it:

  1. Change to the $GROUPER/contrib/load/groupmgr directory
